# 04. 코드 상세 설명서

이 문서는 CrimsonRunner의 핵심 코드가 어떤 역할을 하고, 어떤 함수에서 사용되며, 왜 그런 방식으로 구현되었는지 설명한다. 목적은 단순한 줄 번역이 아니라 코드 구조와 설계 의도를 이해하는 것이다.

## 1. 전체 코드 구조

CrimsonRunner는 작은 2D 게임 엔진 구조 위에 러닝 액션 게임 씬을 올린 형태다.

핵심 흐름은 다음과 같다.

```text
main.cpp
→ Application
→ WindowContext / GraphicsContext / Input 초기화
→ RunnerGameScene 초기화
→ 매 프레임 Update
→ 매 프레임 Render
```

게임의 실제 규칙은 대부분 `RunnerGameScene`에 들어 있고, 그래픽 출력은 `GraphicsContext`, `Material`, `MeshRenderer`, `Shader`가 담당한다.

---

# 2. `main.cpp`

## 역할

`main.cpp`는 프로그램의 시작점이다. Windows 애플리케이션이 실행되면 가장 먼저 호출되는 진입 함수에서 `Application`을 만들고 실행한다.

## 사용되는 곳

- 실행 파일을 시작할 때 한 번 사용된다.
- 게임 루프 자체는 `Application` 내부에서 돌아간다.

## 설계 이유

게임 전체 실행 코드를 `main.cpp`에 직접 작성하지 않고 `Application` 클래스로 넘긴 이유는 역할 분리를 위해서다. `main.cpp`는 시작만 담당하고, 창 생성, 그래픽 초기화, 씬 업데이트 같은 복잡한 과정은 `Application`이 관리한다.

---

# 3. `Application`

## 역할

`Application`은 게임 전체 생명주기를 관리한다.

주요 책임은 다음과 같다.

- 윈도우 생성
- DirectX 그래픽 초기화
- 입력 시스템 갱신
- 씬 초기화
- 게임 루프 실행
- 종료 시 리소스 정리

## 주요 함수

### `Initialize()`

게임 실행에 필요한 시스템을 준비한다.

이 함수에서 일반적으로 다음 작업이 이루어진다.

1. `WindowContext`로 게임 창을 만든다.
2. `GraphicsContext`로 DirectX 렌더링 환경을 만든다.
3. `RunnerGameScene`을 생성하고 초기화한다.

### `Run()`

게임 루프를 실행한다.

게임 루프는 다음 순서로 반복된다.

```text
입력 갱신
→ 시간 계산
→ 씬 Update
→ 화면 지우기
→ 씬 Render
→ 화면 출력
```

### `Shutdown()`

게임 종료 시 그래픽 리소스, 윈도우 리소스 등을 정리한다.

## 설계 이유

게임은 한 번 실행하고 끝나는 프로그램이 아니라 계속 반복되는 프로그램이다. 따라서 반복 실행 구조를 하나의 중앙 관리자에 모아두는 것이 유지보수에 유리하다.

---

# 4. `WindowContext`

## 역할

Windows 운영체제의 창을 생성하고 관리한다.

## 사용되는 곳

`Application::Initialize()`에서 창을 만들 때 사용된다. 생성된 창 핸들(HWND)은 `GraphicsContext`가 DirectX 출력 대상을 만들 때 사용한다.

## 설계 이유

Windows API 코드는 복잡하고 게임 로직과 직접 관련이 없다. 따라서 창 관리 코드를 별도 클래스로 분리하면 게임 로직이 깔끔해진다.

---

# 5. `GraphicsContext`

## 역할

DirectX 11 렌더링 환경을 관리한다.

주요 책임은 다음과 같다.

- DirectX 장치 생성
- SwapChain 생성
- RenderTarget 생성
- 화면 지우기
- 알파 블렌딩 설정
- 디버그 선 그리기
- 텍스트 출력 큐 처리

## 주요 함수

### `Initialize(HWND hwnd, int width, int height)`

DirectX 장치와 SwapChain을 생성한다.

- `device`: GPU 리소스를 만드는 데 사용된다.
- `context`: 실제 렌더링 명령을 보내는 데 사용된다.
- `swapChain`: 그려진 화면을 사용자에게 보여주는 버퍼 교체 장치다.

### `BeginFrame(const Color& clearColor)`

한 프레임의 렌더링을 시작한다.

수행 작업:

1. 이전 프레임의 디버그 선을 비운다.
2. 화면을 지정한 색으로 지운다.
3. RenderTarget을 설정한다.
4. 알파 블렌딩을 켠다.
5. Viewport를 설정한다.

### `EndFrame()`

한 프레임의 렌더링을 끝낸다.

수행 작업:

1. 디버그 선을 그린다.
2. 큐에 쌓인 텍스트를 그린다.
3. SwapChain을 Present해서 화면에 표시한다.

### `QueueText(...)`

텍스트를 바로 그리지 않고 큐에 저장한다. 실제 출력은 `EndFrame()`의 `DrawQueuedText()`에서 이루어진다.

## 설계 이유

DirectX 렌더링은 여러 단계가 필요하다. 이를 `GraphicsContext`에 모아두면 게임 씬은 “무엇을 그릴지”에 집중하고, “어떻게 DirectX에 명령을 보낼지”는 그래픽 시스템이 담당하게 된다.

---

# 6. `Input`

## 역할

키보드 입력 상태를 관리한다.

## 주요 함수

### `IsKeyDown(key)`

현재 키가 눌려 있는지 확인한다.

사용 예:

- 아래 방향키를 누르고 있는 동안 슬라이드 유지

### `WasKeyPressed(key)`

이번 프레임에 새로 눌린 키인지 확인한다.

사용 예:

- 점프 시작
- 게임오버 상태에서 재시작

### `WasKeyReleased(key)`

이번 프레임에 키가 떼어졌는지 확인한다.

사용 예:

- 점프 키를 빨리 떼면 점프 높이를 줄이는 짧은 점프 처리

## 설계 이유

점프와 슬라이드는 입력 성격이 다르다.

- 점프: 눌렀을 때 한 번 발생
- 슬라이드: 누르고 있는 동안 유지

이를 구분하려면 현재 상태뿐 아니라 이전 프레임 상태도 필요하다. 그래서 `Input`은 키의 이전 상태와 현재 상태를 비교한다.

---

# 7. `GameObject`, `Component`, `Transform`

## `GameObject` 역할

게임 월드에 존재하는 하나의 대상이다.

예시:

- 배경
- 플레이어
- 바닥
- 경찰
- 개
- 벽
- 감옥
- 게임오버 오버레이

## `Transform` 역할

오브젝트의 위치와 크기를 저장한다.

- `position`: 화면상 위치
- `scale`: 화면상 크기

## `Component` 역할

오브젝트에 붙는 기능 단위다. 현재 가장 중요한 컴포넌트는 `MeshRenderer`다.

## 설계 이유

오브젝트 자체와 기능을 분리하면 확장성이 좋아진다. 예를 들어 같은 `GameObject`라도 어떤 컴포넌트를 붙이느냐에 따라 화면에 그려지는 물체가 될 수도 있고, 논리 전용 물체가 될 수도 있다.

---

# 8. `MeshRenderer`

## 역할

`GameObject`를 화면에 그리는 컴포넌트다.

## 사용되는 곳

`RunnerGameScene::Initialize()`에서 배경, 플레이어, 장애물, 바닥, 감옥 오브젝트에 붙는다.

## 작동 방식

`MeshRenderer`는 다음 정보를 사용한다.

- 오브젝트의 `Transform`
- 사각형 `Mesh`
- 이미지 또는 색을 가진 `Material`

렌더링 시 `Material`을 바인딩하고, `Mesh`를 그린다.

## 설계 이유

CrimsonRunner의 대부분 시각 요소는 사각형에 PNG 이미지를 입힌 형태다. 따라서 `MeshRenderer` 하나로 배경, 플레이어, 장애물 등 대부분을 처리할 수 있다.

---

# 9. `ResourceManager`

## 역할

게임에서 사용하는 그래픽 리소스를 생성하고 보관한다.

## 주요 함수

### `CreateMaterial(name, color)`

단색 머티리얼을 만든다.

사용 예:

- 게임오버 검은 오버레이
- PNG 로딩 실패 시 임시 색상

### `CreateTexturedMaterial(name, tint, pixels, width, height)`

코드에서 만든 픽셀 배열을 텍스처로 바꾼다.

사용 예:

- PNG 파일을 못 찾았을 때 사용하는 fallback 텍스처

### `LoadMaterialFromFile(name, tint, filePath)`

PNG 파일을 읽어서 머티리얼로 만든다.

사용 예:

- 플레이어 스프라이트
- 낮/밤 배경
- 경찰, 개, 벽, 위쪽 벽
- 게임오버 감옥

## 설계 이유

리소스를 매번 새로 만들면 메모리와 관리가 복잡해진다. `ResourceManager`가 이름을 기준으로 보관하면 여러 오브젝트가 같은 리소스를 안정적으로 사용할 수 있다.

---

# 10. `Material`

## 역할

화면에 그릴 색상과 텍스처 정보를 가진다.

## 주요 함수

### `Create(...)`

기본 머티리얼 정보를 생성한다. 색상과 셰이더를 연결한다.

### `CreateTexture(...)`

메모리에 있는 픽셀 데이터를 DirectX 텍스처로 만든다.

### `LoadTextureFromFile(...)`

PNG 파일을 읽어서 DirectX 텍스처로 만든다. Windows WIC를 사용한다.

### `SetSpriteUV(...)`

스프라이트 시트에서 현재 보여줄 프레임 영역을 설정한다.

플레이어 PNG는 여러 프레임이 한 장에 들어 있으므로, 현재 애니메이션 프레임에 해당하는 UV 좌표를 지정해야 한다.

### `Bind(...)`

렌더링 직전에 머티리얼 정보를 GPU에 전달한다.

## 중요한 구현: Point 필터

```cpp
samplerDesc.Filter = D3D11_FILTER_MIN_MAG_MIP_POINT;
```

픽셀아트는 부드럽게 보간하면 흐릿해진다. Point 필터는 픽셀을 또렷하게 유지한다.

## 설계 이유

색상, PNG, 스프라이트 시트 프레임 정보를 모두 `Material`이 관리하면 렌더링 코드가 단순해진다.

---

# 11. `BasicColor.hlsl`

## 역할

GPU에서 실행되는 셰이더다. 각 픽셀의 최종 색을 계산한다.

## 주요 동작

1. 정점 위치를 월드 행렬로 변환한다.
2. 텍스처를 사용할 경우 PNG 색을 읽는다.
3. 스프라이트 모드면 UV 좌표를 현재 프레임 영역으로 바꾼다.
4. 텍스처 색과 tint 색을 곱한다.
5. 투명도가 낮은 픽셀은 버린다.

## 사용되는 곳

모든 `Material`은 `BasicColor.hlsl` 셰이더를 사용한다.

## 설계 이유

CrimsonRunner는 복잡한 조명이나 3D 효과가 필요 없다. 기본 색상과 텍스처 출력만 있으면 충분하므로 하나의 범용 셰이더로 처리한다.

---

# 12. `RunnerGameScene`

`RunnerGameScene`은 CrimsonRunner의 핵심이다. 게임 규칙 대부분이 여기에 있다.

## 12-1. 멤버 변수 역할

### `ResourceManager resources`

씬에서 사용할 리소스를 관리한다.

### `Player player`

플레이어 상태를 저장한다.

### `SpeedData speedData`

속도와 난이도 증가 규칙을 저장한다.

### `SpawnData spawnData`

장애물 생성 간격과 확률을 저장한다.

### `HitboxData hitboxData`

플레이어 충돌 판정 값을 저장한다.

### `std::vector<Obstacle> obstacles`

현재 활성화된 장애물 목록이다.

### `std::vector<GameObject*> obstacleVisuals`

장애물 표시용 GameObject 풀이다. 장애물을 매번 생성/삭제하지 않고 재사용한다.

### `GameObject* background`

현재 배경 오브젝트다.

### `Material* policeMaterial`, `dogMaterial`, `wallMaterial`, `lowBarMaterial`

장애물 종류별 머티리얼이다.

### `int score`

현재 점수다.

### `GameState gameState`

현재 게임 상태다. `Play` 또는 `GameOver`다.

### `float jailDropOffsetY`

게임오버 감옥이 위에서 얼마나 떨어져 있는지 나타낸다.

## 12-2. `Initialize(GraphicsContext& graphics)`

### 역할

게임 씬에 필요한 모든 오브젝트와 리소스를 준비한다.

### 사용되는 함수

- `ResourceManager::Initialize`
- `LoadMaterialFromFile`
- `CreateObject`
- `AddComponent<MeshRenderer>`
- `ResetGame`

### 설계 이유

게임 시작 전에 모든 주요 오브젝트를 만들어두면 플레이 중 생성 비용을 줄일 수 있다. 장애물도 미리 여러 개 만들어두고 위치와 머티리얼만 바꿔 재사용한다.

## 12-3. `Update(float dt)`

### 역할

매 프레임 게임 상태를 갱신한다.

### 주요 로직

```cpp
if (gameState == GameState::GameOver && Input::WasKeyPressed('R'))
```

게임오버 상태에서만 재시작한다. 플레이 중 R키 오작동을 막기 위한 조건이다.

```cpp
if (gameState == GameState::Play)
```

플레이 중일 때만 점수, 플레이어, 장애물을 업데이트한다.

```cpp
if (CheckOverlap(forgivingBox, obstacle))
```

충돌하면 게임오버로 바꾼다.

### 설계 이유

게임 상태를 기준으로 동작을 분리해야 한다. 플레이 중과 게임오버 중에 가능한 입력과 업데이트가 다르기 때문이다.

## 12-4. `Render(GraphicsContext& graphics)`

### 역할

상태에 따라 UI 텍스트를 화면에 예약한다.

### 사용 방식

- 플레이 중: 점수 표시
- 게임오버: GAME OVER, R RESTART 표시

### 설계 이유

오브젝트 렌더링은 `Scene::Render`가 처리하고, 텍스트 UI는 `RunnerGameScene::Render`에서 상태에 따라 추가한다.

## 12-5. `UpdatePlayer(float dt)`

### 역할

플레이어 입력, 점프, 슬라이드, 중력을 처리한다.

### 주요 입력

```cpp
Space 또는 ↑: 점프
↓: 슬라이드
```

### 점프 처리 이유

점프는 바닥에 있을 때만 시작된다. 공중에서 계속 점프하면 무한 점프가 되므로 `player.isGrounded` 조건을 둔다.

### 짧은 점프 처리 이유

점프 키를 빨리 떼면 상승 속도를 줄인다. 이 방식은 플레이어가 점프 높이를 어느 정도 조절할 수 있게 만든다.

### 슬라이드 처리 이유

슬라이드는 누르고 있는 동안 충돌 박스 높이를 줄인다. 위쪽 벽 장애물을 피하기 위해 필요하다.

## 12-6. `UpdatePlayerVisual()`

### 역할

플레이어의 화면 표시 상태를 실제 상태에 맞게 바꾼다.

### 주요 작업

- 슬라이드 상태면 가로로 길고 낮게 표시한다.
- 달리기/점프 상태면 기본 크기로 표시한다.
- 현재 애니메이션 프레임에 맞게 스프라이트 UV를 바꾼다.

### 설계 이유

플레이어의 충돌 크기와 화면 이미지는 매 프레임 상태에 따라 달라진다. 이를 한 함수에 모아두면 플레이어 표시 로직을 관리하기 쉽다.

## 12-7. `UpdateObstacles(float dt)`

### 역할

장애물을 왼쪽으로 이동시키고, 화면 밖 장애물을 제거하고, 부족한 장애물을 새로 만든다.

### 주요 작업

1. 모든 장애물의 x 위치를 줄인다.
2. 시각 오브젝트 위치를 갱신한다.
3. 화면 밖으로 나간 장애물을 제거한다.
4. 장애물 수가 부족하면 `SpawnNextObstacle()`을 호출한다.

### 설계 이유

러닝 게임에서는 플레이어가 앞으로 달리는 것처럼 보이지만, 실제 구현은 장애물을 왼쪽으로 움직이는 방식이 단순하다.

## 12-8. `SpawnNextObstacle()`

### 역할

다음 장애물의 위치와 종류를 결정한다.

### 주요 계산

- 점프 체공 시간을 계산한다.
- 현재 속도에서 안전하게 넘을 수 있는 최소 간격을 계산한다.
- 점수가 높아질수록 장애물 간격을 줄인다.
- 랜덤으로 장애물 종류를 고른다.
- 일정 점수 이후에는 두 장애물을 연속으로 배치할 수 있다.

### 설계 이유

장애물을 완전 랜덤으로 만들면 피할 수 없는 패턴이 생길 수 있다. 그래서 점프 가능 시간과 속도를 이용해 최소 간격을 보장한다.

## 12-9. `SpawnObstacle(...)`

### 역할

장애물 하나를 실제로 배치한다.

### 중요한 점

이 함수는 충돌 판정 크기와 화면 표시 크기를 분리한다.

```text
Obstacle.width / height
→ 충돌 판정 크기

GetObstacleVisualWidth / Height
→ 화면 표시 크기
```

### 설계 이유

이미지 전체를 충돌 판정으로 쓰면 투명 여백이나 장식 때문에 억울한 충돌이 생긴다. 따라서 실제 판정은 작게, 이미지는 보기 좋게 표시한다.

## 12-10. `GetObstacleVisualPosition(...)`

### 역할

충돌 박스 위치를 기준으로 스프라이트가 자연스럽게 바닥에 붙도록 화면 위치를 계산한다.

### 설계 이유

이미지 높이와 충돌 박스 높이가 다르기 때문에 같은 중심점을 쓰면 이미지가 바닥에서 뜨거나 파묻힐 수 있다. 이 함수는 그 차이를 보정한다.

## 12-11. `CheckOverlap(...)`

### 역할

플레이어와 장애물이 충돌했는지 확인한다.

### 방식

AABB 사각형 충돌이다.

두 조건을 모두 만족하면 충돌이다.

1. 가로 방향으로 겹친다.
2. 세로 방향으로 겹친다.

### 설계 이유

러닝 게임의 장애물 충돌에는 복잡한 물리 엔진이 필요 없다. 사각형 충돌은 빠르고 예측 가능하다.

## 12-12. `GetForgivingPlayerBox()`

### 역할

플레이어 충돌 박스를 약간 줄인다.

### 설계 이유

보이는 픽셀과 실제 판정이 너무 딱 맞으면 플레이어가 억울하게 죽었다고 느낄 수 있다. 약간 작은 판정을 사용하면 게임이 더 공정하게 느껴진다.

## 12-13. `GetGroundSurfaceY()`

### 역할

바닥 표면의 y 좌표를 계산한다.

### 사용되는 곳

- 지상 장애물 y 위치 계산
- 플레이어 발 위치 기준

### 설계 이유

바닥 기준을 여러 곳에 하드코딩하면 나중에 위치가 어긋난다. 하나의 함수에서 계산하면 지면 기준을 통일할 수 있다.

## 12-14. `GetLowBarCenterY(...)`

### 역할

슬라이드로 피해야 하는 위쪽 벽의 높이를 계산한다.

### 설계 이유

위쪽 벽은 점프가 아니라 슬라이드로 피해야 한다. 따라서 서 있는 플레이어 머리에는 걸리고, 슬라이드한 플레이어 머리 위로는 지나가도록 높이를 계산한다.

## 12-15. `UpdateGameOverBars(float dt)`

### 역할

게임오버 연출을 처리한다.

### 주요 작업

- 감옥이 위에서 아래로 떨어진다.
- 검은 오버레이를 화면에 표시한다.
- 기존 막대 오브젝트는 숨긴다.

### 설계 이유

게임오버 상태를 명확하게 보여주기 위해 시각적 피드백을 추가했다. 단순히 텍스트만 띄우는 것보다 감옥 연출이 게임 콘셉트에 맞다.

## 12-16. `ResetGame()`

### 역할

게임을 처음 상태로 되돌린다.

### 초기화 대상

- 플레이어 위치
- 플레이어 속도
- 플레이어 상태
- 점수
- 난이도
- 장애물 목록
- 감옥 위치
- 오버레이 위치
- 배경 상태

### 설계 이유

재시작 시 이전 게임의 상태가 남아 있으면 버그가 생긴다. 따라서 재시작에 필요한 모든 값을 한 함수에서 명확하게 초기화한다.

# 13. 에셋 로딩 실패 대비 코드

CrimsonRunner는 PNG 파일을 우선 사용하지만, 일부 PNG를 못 찾았을 때 코드 생성 텍스처를 대신 사용한다.

## 장점

- 에셋 파일이 누락되어도 게임이 완전히 멈추지 않는다.
- 개발 중 임시 확인이 가능하다.

## 단점

- 코드 안에 오래된 텍스처 생성 코드가 남아 파일이 길어진다.

## 개선 방향

최종 제출용이라면 fallback 코드를 유지해도 된다. 다만 상용 수준으로 정리한다면 fallback 텍스처는 별도 파일이나 디버그 전용 코드로 분리하는 것이 좋다.

# 14. 최종 설계 판단

현재 구조는 작은 프로젝트에는 적합하다.

좋은 점:

- 게임 루프와 씬 로직이 명확하다.
- 입력, 그래픽, 리소스 관리가 분리되어 있다.
- PNG 에셋을 쉽게 교체할 수 있다.
- 충돌 크기와 시각 크기를 분리해 플레이 감각을 조정할 수 있다.

아쉬운 점:

- `RunnerGameScene.cpp`에 너무 많은 책임이 모여 있다.
- 플레이어, 장애물, 점수, 게임오버 연출을 나중에는 클래스로 분리하는 것이 좋다.
- 장애물 데이터는 코드 하드코딩보다 설정 파일이나 테이블로 관리하면 더 좋다.

현재 단계에서는 기능 완성과 설명 가능성을 우선한 구조로 볼 수 있다.
